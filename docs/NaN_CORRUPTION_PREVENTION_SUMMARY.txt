┌─────────────────────────────────────────────────────────────────────────┐
│ File: organize_files_array_with_seasons.js                             │
│ Line: 34                                                                │
│                                                                         │
│ DANGEROUS CODE:                                                         │
│   return parseInt(a) - parseInt(b);                                     │
│                                                                         │
│ PROBLEM:                                                                │
│   When 'a' = "Featurettes" or "Specials"                              │
│   parseInt("Featurettes") = NaN                                        │
│   NaN - [any number] = NaN                                             │

WHY IT HAPPENED:
• TV shows have non-numeric season keys: "Featurettes", "Specials", "Extras"
• Code assumed ALL season keys were numeric
• No validation on parseInt() results
• No checks before arithmetic operations
• No validation before writing JSON

💡 KEY LESSONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. NEVER assume object keys are numeric
   • TV shows have: "Featurettes", "Specials", "Extras"
   • ALWAYS filter before parsing

2. parseInt() can fail silently
   • Returns NaN for non-numeric strings
   • NaN spreads through arithmetic operations
   • ALWAYS validate with isNaN()

3. JavaScript quirks with NaN:
   • NaN !== NaN (NaN is not equal to itself!)
   • typeof NaN === 'number' (Yes, really!)
   • NaN || 0 === 0 (Falsy coercion works)
   • But direct assignment can still write NaN

4. Sort comparators must return numbers
   • If comparator returns NaN, behavior is undefined
   • Different JS engines handle it differently
   • Can cause unpredictable sorting

5. JSON.stringify preserves NaN as literal string
   • JSON.stringify({val: NaN}) → '{"val":NaN}'
   • This creates INVALID JSON
   • Must be caught before writing
